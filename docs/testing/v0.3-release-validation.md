# v0.3 release validation

v0.3 is validated as a deterministic policy release, not as a claim that every
coding task benefits from compression.

## Accepted fixture contract

The replay corpus covers:

- ANSI and non-semantic control cleanup inside tool-result content;
- runs of three or more identical output lines;
- consecutive identical tool-result messages with the same tool-call ID;
- identical tool results separated by an assistant turn;
- ordinary user and assistant content;
- a tool output exceeding the configured policy input limit;
- the global kill switch.

The fixture must retain user/assistant content, tool-call identity, each unique
line, the exact repetition count, and separated tool results. It must show
positive literal and net reduction when eligible. Oversized input and killed or
ineligible policies must be identity functions.

## Protocol evidence

The fake-provider suite verifies:

- observe mode continues to forward original body bytes;
- optimize mode forwards the immutable prepared request only when a policy
  changes content;
- active decisions include policy/version, reason, affected items, and removed
  characters;
- token reduction is calculated once from baseline versus final forwarding;
- optimization tokens and policy retries remain explicit;
- no transform creates an inference retry.

## Release command

```sh
pnpm check
pnpm build
pnpm --filter @token-shuffle/web test:e2e
```

Live OpenCode and Pi optimize-mode compatibility remains provisional until
policy-enabled sessions are recorded separately from their verified observe-mode
results.
